## Implementation order

### Step 1 — dependencies & type scoping (foundational; do first)

1. **`apps/client/package.json`** — add to `devDependencies` (keep alphabetized; SAME version as `apps/server` to avoid a second `@types/node` in the workspace): `"@types/node": "^20.17.0"`.
2. **Regenerate the lockfile** — `corepack pnpm install` at repo root. Symlinks `apps/client/node_modules/@types/node`, updates `pnpm-lock.yaml`. Commit the lockfile.
3. **`apps/client/tsconfig.app.json`** — add `"types": []` to `compilerOptions`.
   **Why (Phase-1 verdict b):** `tsconfig.app.json` (compiles `src`) inherits NO `types` array, so it auto-includes every `@types/*` in scope. Once `@types/node` exists it would pull Node's ambient globals (`URL`, `fetch`, `setTimeout`→`NodeJS.Timeout`, `Buffer`, `process`) into the DOM app compilation and shadow DOM-lib types. `"types": []` disables ambient `@types` auto-inclusion for `src`; `vite/client` (via `vite-env.d.ts` triple-slash) and `@types/react`/`react-dom` (import-resolved) are NOT gated by `types` and keep working. `tsconfig.node.json` has no `types` array → auto-includes `@types/node` for `vite.config.ts`, no change. (Precedent: `apps/server/tsconfig.json` uses `"types": ["node"]`.)

### Step 2 — vite.config.ts

Add above `defineConfig` (semicolons, matching this file's style):

```ts
import { execSync } from 'node:child_process';
import { readFileSync } from 'node:fs';
import { fileURLToPath } from 'node:url';

// CF 54: telemetry clientVersion = pkg.version + short git SHA
// (tech-architecture.md § 8.3 — M1 internal builds are tagged by commit SHA).
// Fallback 'local' keeps builds working in any environment without git.
const configDir = fileURLToPath(new URL('.', import.meta.url));
const pkgVersion: string = JSON.parse(
  readFileSync(new URL('./package.json', import.meta.url), 'utf-8'),
).version;
let gitSha = 'local';
try {
  // stdio silences git's stderr so a git-less build's error line
  // ('not a git repository') does not leak into build output before the catch.
  gitSha = execSync('git rev-parse --short HEAD', {
    cwd: configDir,
    stdio: ['ignore', 'pipe', 'ignore'],
  })
    .toString()
    .trim();
} catch {
  // git absent (e.g. a tarball build) — keep the 'local' fallback.
}
const clientVersion = `${pkgVersion}+${gitSha}`;
```

Then add a sibling key inside the exported config object:

```ts
  define: {
    __CLIENT_VERSION__: JSON.stringify(clientVersion),
  },
```

## Edge cases

1. **`@types/node` is required AND must be scoped (rewritten).** `tsc -b` typechecks `vite.config.ts` (via `tsconfig.node.json`); its new `node:` imports + `URL`/`import.meta.url` need `@types/node`, which `apps/client` does not declare (only `apps/server` does). On this dev machine an ambient `C:/Users/trobbins/node_modules/@types/node` false-greens the local typecheck; a clean CI checkout fails `TS2307`/`TS2304`/`TS2339`. Fix = add `@types/node` to `apps/client` **and** `"types": []` on `tsconfig.app.json` (Step 1). The Step-3 `declare const` covers only `__CLIENT_VERSION__` in the `src` project — it does nothing for the node builtins in the node project.
2. **The typeof guard is not dead code under Vite** — esbuild substitutes the identifier → `typeof "0.0.1+abc" !== 'undefined'` → constant-folds. Don't simplify it away.
3. **Do NOT use `VITE_*` env vars or `import.meta.env`** — the repo reserves env conventions (decision-log 2026-05-26 § "M1.5d Phase-1 canon reconciliation"); a `define` constant is the narrowest tool.
4. **Server fixtures keep `'m1.5c-pr1'`** — self-contained test literals, valid under `min(1)`, decoupled from the client constant. Leaving them is correct.
5. **OneDrive path + spaces:** `execSync` with `cwd: configDir` is safe (no shell path interpolation); the `stdio` array suppresses git's stderr. Don't build a shell string containing the path.
6. **Windows dev quirk:** repo under OneDrive; pnpm via `corepack pnpm`. A stray postcss-config complaint from `vite build` is the known OneDrive-parent-dir leak (tech-architecture.md § 8.4) — pre-existing.
7. **PostHog continuity:** after this ships, dashboard filters on `clientVersion = 'm1.5c-pr1'` stop matching new events. That's the point — note it in the closing entry so the changeover isn't read as a data gap.
8. **Schema-field precision.** The two schema files (`content-schemas.ts:805` + byte-identical mirror `packages/content/src/schemas.ts:805`) DEFINE the `clientVersion` wire field (`readonly clientVersion: string`). Value-only change → no schema edit, schema-sync gate trivially green. "No schema change" means the field definition is untouched, not that schemas are unrelated to `clientVersion`.
9. **Local verification caveat.** See Step 6 — the ambient `@types/node` leak means a green local `typecheck` does NOT prove the fix; CI is the authoritative gate.
